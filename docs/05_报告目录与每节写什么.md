## 1. Introduction

### 这一节写什么
这一节写 3 件事：
1. 这个作业是什么；
2. 你们修复了什么类型的问题；
3. 你们后续打算研究什么实验问题。

### 建议内容
- 简述任务：QANet on SQuAD v1.1，目标是预测答案 span 的起止位置。
- 说明代码仓库一开始包含大量故意埋设的 bug。
- 说明你们的工作分为两个部分：
  - 修复 pipeline 和 mechanism bug；
  - 基于修复后的系统做实验研究。
- 简要列出 3 个研究问题 / 假设。

### 一段可直接改写的模板
> This assignment focuses on debugging and experimentally analyzing a PyTorch implementation of QANet for extractive question answering on SQuAD v1.1. The provided repository contains both pipeline-level and mechanism-level implementation bugs, which prevent reliable training and evaluation. Our work first repaired the full training and evaluation pipeline, then corrected multiple deep learning components including loss computation, normalization, optimization, scheduling, attention, and convolution-related modules. Based on the corrected framework, we further conducted three controlled experiments to study how optimizer choice, normalization strategy, and learning-rate scheduling affect training stability and generalization performance.

## 2. Debugging Analysis

### 这一节写什么
这是报告里的大头。重点不是“我改了哪些文件”，而是：
- bug 是什么；
- 为什么错；
- 它会导致什么训练/评估问题；
- 你们如何修复；
- 修复后系统行为如何改善。

### 推荐结构
#### 2.1 Stage I: Fundamental Pipeline Fixes
可放这些：
- 训练入口与参数封装错误
- backward / optimizer / checkpoint 保存问题
- evaluate checkpoint 读取问题
- notebook 路径和依赖问题
- download / Colab executability 问题

#### 2.2 Stage II: Deep Learning Mechanism Fixes
可放这些：
- loss
- dropout / activation
- embedding / conv / pointer head
- positional encoding / attention / encoder block
- normalization
- optimizers
- schedulers
- initialization

### 这一节最重要的写法原则
- 不要只列 patch
- 要写“错误机制 -> 行为影响 -> 修复逻辑”
- 能结合训练现象就结合，比如：loss 不下降、shape error、checkpoint 无法保存、metrics 异常

### 推荐句式
- “This bug prevented the pipeline from ...”
- “Although the code could still run, the implementation was theoretically incorrect because ...”
- “This error distorted the training dynamics by ...”
- “After correction, the component became consistent with the intended formulation of ...”

### 可直接引用的素材来源
- `docs/02_Bug清单与报告素材.md`

## 3. Experimental Setup

### 这一节写什么
把所有实验共享的设置集中写，避免后面重复。

### 应该包含
- Dataset：mini train + full dev
- Preprocessing settings：`para_limit=400`, `ques_limit=50`
- Model family：修复后的 QANet
- Shared hyperparameters：batch size、seed、loss、训练步数
- Evaluation metrics：loss / F1 / EM
- Environment：本地 + Colab 验证

### 建议写法
- 说明为什么使用同一套 protocol：保证实验可比性。
- 说明所有实验只改变一个主要变量。

### 模板
> All experiments were conducted on the repaired QANet implementation using the mini training subset and the full SQuAD v1.1 development set provided in the assignment resources. Unless otherwise stated, we fixed the preprocessing limits to `para_limit=400` and `ques_limit=50`, used batch size 8, seed 42, and the QA negative log-likelihood loss. Performance was evaluated using validation loss, F1, and Exact Match (EM), so that all experimental comparisons were made under a consistent protocol.

## 4. Experiment 1 — Optimizer

### 这一节写什么
- Research Question
- Hypothesis
- Setup
- Results
- Analysis

### 结构模板
#### Research Question
不同优化器是否会显著影响当前修复版 QANet 的收敛速度和 dev performance？

#### Hypothesis
我们假设 `adam` 收敛更快，但 `sgd` 可能更稳定 / 泛化更稳（或相反，取决于你们实际结果）。

#### Setup
- Baseline: SGD
- Variant: Adam
- 控制变量：其余参数不变

#### Results
插入表格。

#### Analysis
- 结果支持还是不支持假设？
- 更快下降是否转化成更高 F1/EM？
- 是否更不稳定？

- 哪种对小 batch 更合适？
- 哪种在 dev F1/EM 上更好？

## 6. Experiment 3 — Scheduler / Regularization

### 这一节写什么
如果你们做 scheduler：
- 比较 `step` vs `cosine`

如果你们做 dropout：
- 比较 `dropout=0.1` vs `0.2` / `0.05`

### 分析重点
- 收敛曲线是否更平滑？
- dev 指标是否改善？
- 是否说明学习率控制 / 正则化强度影响了泛化？

## 7. Discussion

### 这一节写什么
这一节不是重复结果，而是把“debugging”和“experiments”联系起来。

### 建议回答的问题
- 哪类 bug 最致命？为什么？
- 哪些 bug 会导致 runtime failure，哪些 bug 更隐蔽但会扭曲训练？
- 修复后最明显的训练变化是什么？
- 实验结果说明了什么机制问题？
- 你们的实验有哪些局限性？

### 很适合写的内容
- “代码能运行”不等于“机制正确”
- 很多 scheduler / optimizer / normalization bug 不一定立刻崩，但会破坏结果解释性
- 本次作业的价值在于把工程调试和机制理解结合起来

## 最后提醒

### 报告一定要满足 PDF 的 3 个隐含要求
1. **完整性**：不要只写几个 major bugs。
2. **分析性**：不要只说“改了什么”，要说“为什么错、影响什么”。
3. **实验性**：实验不是调参流水账，而是要有 hypothesis 和 analysis。
